# EventSnap — Frontend Flows (MVP)

Документ для фронтенда: как пользовательские сценарии раскладываются на вызовы API.
Актуальный контракт эндпоинтов: `docs/api.md`.
Важно: часть методов ниже еще в статусе `запланирован`; проверяй статус в `docs/api.md`.

## Базовые правила интеграции

- Все API-вызовы идут в JSON.
- Для защищенных методов передавать заголовок `Authorization: Bearer <access_token>`.
- При `401` на защищенном запросе один раз выполнить `POST /auth/refresh`, затем повторить исходный запрос.
- Если refresh неуспешен (`401`), разлогинить пользователя.

## 1) Регистрация пользователя

### Пользовательский сценарий

Пользователь вводит email, пароль, имя и нажимает "Зарегистрироваться".

### API-последовательность

1. `POST /auth/register`
2. Сохранить `access_token` и `refresh_token`
3. Перейти в экран "Мои события" или профиль

### Пример запроса

```json
{
  "email": "user@example.com",
  "password": "strong-password-123",
  "display_name": "Alex"
}
```

### Что важно обработать

- `409` — email уже занят
- `422` — ошибка валидации формы

## 2) Вход пользователя

### Пользовательский сценарий

Пользователь вводит email и пароль, нажимает "Войти".

### API-последовательность

1. `POST /auth/login`
2. Сохранить `access_token` и `refresh_token`
3. Загрузить данные пользователя `GET /auth/me`
4. Перейти на главный экран

### Что важно обработать

- `401` — неверный email/пароль

## 3) Автообновление токена (фон)

### Когда

Любой защищенный запрос вернул `401`.

### API-последовательность

1. `POST /auth/refresh` с текущим `refresh_token`
2. Обновить токены в хранилище
3. Повторить исходный запрос
4. Если refresh неуспешен (`401`) — очистить сессию и открыть экран логина

## 4) Просмотр своего профиля

### API-последовательность

1. `GET /auth/me`

### UI-данные

- `id`, `email`, `display_name`, `avatar_s3_key`, `role`

## 5) Обновление профиля

### Пользовательский сценарий

Пользователь меняет отображаемое имя.

### API-последовательность

1. `PATCH /users/me`
2. Обновить локальное состояние профиля

## 6) Загрузка аватара пользователя

### Пользовательский сценарий

Пользователь выбирает фото аватара.

### API-последовательность

1. `POST /users/me/avatar/upload-url` (получить presigned URL)
2. `PUT <upload_url>` напрямую в S3 с тем же `Content-Type`
3. `POST /users/me/avatar/complete` (подтвердить загрузку)
4. Обновить профиль через `GET /auth/me` или локально

## 7) Создание мероприятия

### Пользовательский сценарий

Организатор заполняет форму события и нажимает "Создать".

### API-последовательность

1. `POST /events`
2. Получить `event_id` из ответа
3. Перейти на экран события

### Что важно обработать

- `422` — невалидные данные формы (например, `ends_at <= starts_at`)

## 8) Список мероприятий пользователя

### Пользовательский сценарий

Экран "Мои события": вкладки "Организую", "Участвую", "Все".

### API-последовательность

1. `GET /events?role=organizer|participant|all&limit=...&offset=...`

## 9) Открытие карточки мероприятия

### API-последовательность

1. `GET /events/{event_id}`

### Что важно обработать

- `404` — событие удалено или не найдено
- `403` — пользователь не является участником события

## 10) Получение QR/ссылки приглашения

### Пользовательский сценарий

Организатор открывает экран "Пригласить участников".

### API-последовательность

1. `GET /events/{event_id}/join-link`
2. Фронтенд генерирует QR по возвращенной ссылке

## 11) Вход в мероприятие по QR

### Пользовательский сценарий

Гость сканирует QR и открывает ссылку вида `...?qr_token=...`.

### API-последовательность

1. Если пользователь не авторизован, показать экран регистрации/логина.
2. После успешной авторизации выполнить `POST /events/join` с `qr_token`.
3. Выполнить `GET /events/{event_id}`.
4. Выполнить `GET /events/{event_id}/gallery`.
5. Выполнить `GET /events/{event_id}/photos`.

### Пример запроса

```json
{
  "qr_token": "a1b2c3d4..."
}
```

## 12) Просмотр галереи мероприятия

### Пользовательский сценарий

Участник открывает галерею события.

### API-последовательность

1. `GET /events/{event_id}/gallery` (настройки галереи)
2. `GET /events/{event_id}/photos?limit=...&offset=...` (только одобренные фото)
3. При скролле догружать следующую страницу

## 13) Загрузка фото в галерею

### Пользовательский сценарий

Участник выбирает фото в браузере и нажимает "Загрузить".

### API-последовательность

1. `POST /events/{event_id}/photos/upload-url` с метаданными файла
2. `PUT <upload_url>` напрямую в S3
3. Обновить галерею или локальный UI-стейт.
4. Если модерация включена, показать статус "на проверке".
5. Если модерация выключена, фото появится в `GET /events/{event_id}/photos`.

### Важная деталь

`Content-Type` в `PUT` должен совпадать с тем, что передали при запросе upload URL.

## 14) Удаление фото

### Пользовательский сценарий

Пользователь удаляет свое фото из галереи.

### API-последовательность

1. `DELETE /photos/{photo_id}`
2. Удалить фото из UI или перезагрузить страницу галереи

### Права

- Автор фото может удалить свое фото
- Организатор может удалить любое фото в своем событии

## 15) Модерация фото (организатор)

### Пользовательский сценарий

Организатор открывает очередь модерации и подтверждает/отклоняет фото.

### API-последовательность

1. `GET /events/{event_id}/photos/pending/count` (бейдж)
2. `GET /events/{event_id}/photos/pending` (список)
3. Для фото, которое нужно одобрить, вызвать `POST /photos/{photo_id}/approve`.
4. Для фото, которое нужно отклонить, вызвать `POST /photos/{photo_id}/reject`.

### Пример отклонения

```json
{
  "comment": "Фото не относится к событию"
}
```

## 16) Включение/выключение модерации

### Пользовательский сценарий

Организатор меняет настройку "Модерация фото".

### API-последовательность

1. `PATCH /events/{event_id}/gallery` с `moderation_enabled=true|false`
2. Обновить состояние переключателя в UI

## 17) Загрузка обложки мероприятия

### Пользовательский сценарий

Организатор выбирает обложку события.

### API-последовательность

1. `POST /events/{event_id}/cover/upload-url`
2. `PUT <upload_url>` напрямую в S3
3. `POST /events/{event_id}/cover/complete`
4. Обновить карточку события

## 18) Выход из аккаунта

### Пользовательский сценарий

Пользователь нажимает "Выйти".

### API-последовательность

1. API-вызов не требуется
2. Очистить `access_token` и `refresh_token` в клиенте
3. Перейти на экран логина
